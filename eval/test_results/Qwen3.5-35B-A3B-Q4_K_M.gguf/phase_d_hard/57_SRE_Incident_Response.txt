## 1. Executive Summary
On Monday at 2:00 AM, an automated certificate rotation process deployed an invalid TLS certificate to the API Gateway. This triggered a cascade of failures: an auto-scaling storm, Kubernetes cluster saturation, and a complete loss of observability. The service remained unavailable for 150 minutes, significantly exceeding the monthly SLA allowance. The root cause was a configuration error in the certificate rotation script that was not caught by pre-deployment validation.

---

## 2. Cascade Failure Map
The incident did not occur in isolation. The initial certificate error triggered three distinct secondary failures that compounded the outage.

```mermaid

    I -->|Protection Mechanism| J[DB Enters Read-Only Mode<br/>2:20 AM]
```

**The 3 Secondary Failures:**
1.  **Scaling Storm (2:05 AM):** Health check failures triggered the Horizontal Pod Autoscaler (HPA) to spin up new instances. Since the new instances inherited the bad certificate, they immediately failed health checks, triggering *more* scaling.
2.  **Cluster Saturation (2:10 AM):** The unbounded scaling hit the Kubernetes cluster's hard limit (500 pods). The control plane became unresponsive, preventing any further orchestration or manual intervention.
3.  **Observability Blackout (2:11 AM):** The monitoring system, running on the same cluster, was evicted due to the pod limit. All internal alerts ceased, leaving the team blind to the outage until external customer complaints arrived.

## 3. Root Cause Analysis (RCA)

**Technical Root Cause:**
The automated certificate rotation script (`rotate-cert.sh`) was configured to use a wildcard pattern `*.example.com` instead of the required `*.api.example.com`. This configuration error was introduced during a recent update to the rotation logic.

**Process Root Cause:**
There was no **pre-flight validation** step in the CI/CD pipeline for certificate rotation. The script was allowed to deploy to production without verifying that the certificate CN/SAN matched the active ingress rules. Additionally, the monitoring system was co-located on the same cluster as the production workloads, creating a single point of failure for observability.

---

## 4. The 5 Whys

1.  **Why did API requests fail?**
    *   Because the API Gateway presented an invalid TLS certificate, causing clients to reject the connection.
2.  **Why was the certificate invalid?**
    *   Because the automated rotation script deployed a certificate with the wrong domain (`*.example.com` instead of `*.api.example.com`).
3.  **Why did the script deploy the wrong certificate?**
    *   Because the configuration variable for the target domain was not updated in the rotation script during the last refactor.
4.  **Why was the configuration error not caught before deployment?**
    *   Because the CI/CD pipeline lacked a validation step to verify certificate SANs against the expected ingress domains prior to rotation.
5.  **Why did the outage last 2.5 hours instead of minutes?**
    *   Because the cascade (Scaling Storm + Cluster Saturation + Monitoring Blackout) prevented the team from detecting and mitigating the issue until customers reported it.

## 7. Lessons Learned

1.  **Automation requires Guardrails:** Fully automated processes (like cert rotation) must have validation steps that prevent bad data from propagating to production.
2.  **Observability Independence:** Critical infrastructure (like monitoring) should never share the same resource constraints as the services it is monitoring.
3.  **Scaling Safety:** Auto-scaling policies must account for the health of the *new* instances, not just the failure of the old ones, to prevent infinite loops.
4.  **Detection Gap:** Relying solely on internal monitoring creates a blind spot. External synthetic monitoring or customer feedback loops are essential backups for incident detection.
