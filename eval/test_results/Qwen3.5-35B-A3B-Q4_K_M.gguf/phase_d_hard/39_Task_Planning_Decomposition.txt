Project: Build a real-time dashboard

### 2. Circular Dependency Identification

**The Cycle:**
There is a **Hard Circular Dependency** among Tasks A, B, and C.
1.  **Task A** cannot start until **Task C** is ready (to send emails).
2.  **Task C** cannot start until **Task B** is ready (to know events).
3.  **Task B** cannot start until **Task A** is ready (to validate sources).

**Path:** `A → C → B → A`

**Impact:** If followed literally, the project cannot begin. No backend work can start on A, B, or C without the others being finished, creating a deadlock.

---

### 3. Resolution Strategy

To break the cycle, we must decouple the dependencies during the **Development Phase** while maintaining the logical flow for the **Production Phase**.

**Proposed Resolution: Contract-First & Mocking**

1.  **Define Interfaces First (Week 1):**
    *   Define the API contracts between A, B, and C before writing code.
    *   *Example:* Task C defines an "Event Schema." Task B agrees to emit events in that schema. Task A agrees to call a standard "Email Service" interface.

2.  **Mocking for Development:**
    *   **Task A (Auth):** Implement a **Mock Email Provider** (e.g., logs to console or saves to a file) instead of waiting for Task C. This allows Auth to be built and tested independently.
    *   **Task C (Notification):** Implement a **Mock Data Listener** that accepts predefined JSON payloads instead of waiting for the live Data Pipeline (Task B).
    *   **Task B (Data):** Implement a **Mock Auth Validator** that accepts all requests during development, switching to the real Auth API once Task A is deployed.

3.  **Architectural Decoupling (Production):**
    *   Move Task C (Notifications) to an **Event Bus** (e.g., Kafka/RabbitMQ) rather than a direct API call from Task B. Task B publishes events; Task C subscribes. This removes the direct "Task B must be done before Task C" dependency.
    *   Task A (Auth) should treat Task C as an optional dependency for the "Onboarding" flow, not the core "Login" flow.

*   **Week 8 (Deployment):** Deployment is the final step, but the services (A, B, C) are deployed individually as microservices, satisfying the requirement to deploy them separately.
*   **QA:** QA is involved from Week 1 (planning) to Week 7 (testing), ensuring the cycle break doesn't introduce regressions.
