### 2. Root Cause

The root cause of the incident was a **configuration error in the automated certificate rotation process**. The script was configured to request a certificate with the domain `*.example.com` instead of the correct domain `*.api.example.com`. This incorrect certificate caused TLS handshake failures for all API requests.  The automated nature of the process amplified the impact, rapidly scaling the problem before manual intervention could occur.

---

### 3. 5 Whys

1. **Why did the API requests start failing?** Because the new certificate did not match the domain used by the API gateway.
2. **Why did the new certificate not match the domain?** Because the certificate rotation script was configured with the wrong domain name.
3. **Why was the script configured with the wrong domain name?** Because the configuration was manually updated without proper validation or testing.
4. **Why was there no validation or testing of the configuration change?** Because there was no automated validation step in the certificate rotation process.
5. **Why was there no automated validation step?** Because the initial implementation prioritized speed of deployment over robust error prevention and lacked sufficient safeguards.

### 4. SLA Impact Calculation

* **Monthly Allowed Downtime:** 21.9 minutes

### 5. Blameless Action Items

This incident was not caused by individual error, but by systemic failures in our processes.  These action items focus on preventing recurrence.

**1. Implement Automated Certificate Validation (High Priority - Owner: Platform Engineering)**

*   **Description:**  Add a validation step to the certificate rotation process that automatically checks the newly issued certificate against the expected domain name *before* deploying it to the API gateway. This could involve DNS lookups, connection tests to internal services, or a dedicated validation service.
*   **Timeline:** Within 2 weeks.

**2. Enhance Monitoring and Alerting (High Priority - Owner: SRE)**

*   **Description:**  Improve monitoring to detect certificate errors *before* they impact all API requests.  Specifically, monitor TLS handshake success rates and add alerts for significant drops.  Investigate why the monitoring system itself failed during the incident and implement redundancy/failover mechanisms.
*   **Timeline:** Within 1 week.

**3. Review and Improve Auto-Scaling Configuration (Medium Priority - Owner: DevOps)**

*   **Description:**  Re-evaluate the auto-scaling configuration to prevent runaway scaling in the event of a widespread failure. Consider implementing a maximum pod limit that is more conservative and/or adding a delay to scaling events to allow for manual intervention.
*   **Timeline:** Within 3 weeks.

**4. Strengthen Change Management Process (Medium Priority - Owner: Platform Engineering & Security)**

*   **Description:**  Implement a more rigorous change management process for critical infrastructure configurations like certificate rotation. This should include peer review, automated testing, and a rollback plan.  Consider using infrastructure-as-code to track changes and facilitate rollbacks.
*   **Timeline:** Within 4 weeks.

**5. Database Connection Pool Management (Low Priority - Owner: Database Team)**

*   **Description:**  Investigate options for more resilient database connection pool management, such as dynamic pool sizing or connection limiting to prevent exhaustion during scaling events.
*   **Timeline:** Within 6 weeks.

**6. Post-Incident Review Automation (Low Priority - Owner: SRE)**

*   **Description:** Automate the collection of relevant logs and metrics immediately following an incident to streamline the postmortem process.
*   **Timeline:** Within 8 weeks.
