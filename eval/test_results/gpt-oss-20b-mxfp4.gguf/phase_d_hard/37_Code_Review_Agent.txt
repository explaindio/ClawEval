## The **critical bug** – lost money due to race conditions

> **The `transfer` method is not atomic and is not protected by any lock.**  
> When two threads call `transfer` on the *same* source account (or on two accounts that both target the same destination) they can interleave in a way that causes the balance to be updated incorrectly, effectively “losing” money.

### What can happen

```text
Thread A:   read balance = 1000
Thread B:   read balance = 1000
Thread A:   balance -= 200  → 800
Thread B:   balance -= 200  → 800   (uses the old 1000 value!)
Thread A:   write balance = 800
Thread B:   write balance = 800   (overwrites Thread A’s change)
```

The net result is a deduction of only 200 instead of 400, so 200 dollars disappears.

The same problem occurs on the target side if two threads add to the same account concurrently.

### Why it’s hard to spot

The code *looks* fine – it checks the balance, subtracts, adds, and records a transaction.  
But because there is **no lock** around the whole operation, the read‑modify‑write sequence is not atomic.  
The hint was right: it’s not a float‑precision bug; it’s a concurrency bug.

### Fix

Wrap the entire transfer in a lock that covers **both** accounts:

```python
def transfer(self, target_account, amount):
    amount = Decimal(amount)  # use Decimal everywhere
    if amount <= 0:
        raise ValueError("Amount must be positive")

    # Acquire locks in a consistent order to avoid deadlocks
    first, second = (self, target_account) if id(self) < id(target_account) else (target_account, self)

    with first.lock:
        with second.lock:
            if self.balance < amount:
                return False
            self.balance -= amount
            target_account.balance += amount
            # record transactions on both sides
            self.transactions.append({...})
            target_account.transactions.append({...})
            return True
```

Now the balance updates are atomic and no money can be lost.

### Quick checklist for a production‑ready bank account class

- Use `Decimal` for all monetary values.
- Validate input types (`Decimal`, `str`, `int`).
- Round balances consistently (or rely on `Decimal`).
- Protect all shared mutable state (`balance`, `transactions`) with a lock.
- Record transactions on **both** source and target accounts.
- Avoid deadlocks by acquiring locks in a deterministic order.
- Consider using a dedicated transaction manager or database ACID guarantees instead of manual locking.

Fixing the five obvious issues and the critical concurrency bug will make the class safe, auditable, and free from money‑loss bugs.
