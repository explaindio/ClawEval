## 2. Root Cause  

**Primary Root Cause:**  
The automated certificate rotation script was mis‑configured to issue a wildcard certificate for `*.example.com` instead of the intended `*.api.example.com`. This mis‑configuration was not caught because:

1. **No automated cert validation** – the rotation job did not verify the domain in the issued cert before publishing it to the gateway.
2. **No pre‑deployment health check** – the new cert was not tested in a staging environment or against a test API endpoint.
3. **Monitoring shared the same cluster** – when the cluster became saturated, the monitoring stack lost connectivity, eliminating the alerting signal that could have triggered a faster response.

**Contributing Factors:**

- Lack of a **certificate linting step** in the CI/CD pipeline.
- **Insufficient change‑management review** – the config file containing the domain was edited without a formal approval process.
- **Auto‑scaling loop** had no circuit breaker to stop scaling when health checks were failing for the same reason.
- **Single point of failure** – monitoring and API gateway lived in the same Kubernetes namespace.

## 3. 5 Whys Analysis  

| # | Question | Answer |
|---|----------|--------|
| 1 | **Why** did the API gateway start returning TLS errors? | The certificate installed on the gateway was invalid for the requested domain. |
| 2 | **Why** was the certificate invalid? | The rotation script issued a wildcard cert for `*.example.com` instead of `*.api.example.com`. |
| 3 | **Why** did the rotation script issue the wrong cert? | The configuration file used by the script contained an outdated domain value that had not been updated during the last change. |
| 4 | **Why** was the configuration file outdated? | The change‑management process did not flag the file as a critical change, so it was edited without a formal review or automated linting. |
| 5 | **Why** was there no automated linting or validation? | The CI/CD pipeline for cert rotation lacked a step to parse the issued cert and verify its SANs against the expected domain list. |

## 6. Summary & Lessons Learned  

- A single mis‑configured cert triggered a chain reaction that disabled the entire API, monitoring, and database layers.  
- The lack of automated validation and a shared monitoring environment amplified the impact.  
- The incident demonstrates the importance of *defensive automation* (linting, health checks), *circuit breakers* for auto‑scaling, and *redundant monitoring* to avoid single points of failure.  

By implementing the above action items, we will reduce the risk of a similar cascade and bring our uptime back within SLA bounds.
