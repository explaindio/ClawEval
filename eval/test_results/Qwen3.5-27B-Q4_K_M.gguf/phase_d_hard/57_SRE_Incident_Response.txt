## 5. Root Cause Analysis

**Root Cause:** The automated certificate rotation pipeline utilized a configuration variable that defaulted to the root domain wildcard (`*.example.com`) instead of the specific API subdomain wildcard (`*.api.example.com`).

**Contributing Factors:**
1.  **Lack of Validation:** The pipeline did not validate the Subject Alternative Names (SANs) of the generated certificate against the expected domain list before deployment.
2.  **Coupled Infrastructure:** The monitoring stack was hosted on the same Kubernetes cluster as the application. When the cluster became unstable, the "eyes" of the system went blind.
3.  **Aggressive Auto-Scaling:** The auto-scaler was configured to scale up on health check failures without a circuit breaker, creating a feedback loop that exhausted cluster resources.

---

## 6. The 5 Whys

1.  **Why did the API Gateway fail?**
    *   The SSL certificate presented to clients did not match the requested domain (`*.example.com` vs `*.api.example.com`).
2.  **Why was the wrong certificate deployed?**
    *   The automated rotation script used a hardcoded wildcard pattern that was too broad for the specific API environment.
3.  **Why did the script use the wrong pattern?**
    *   There was no pre-deployment validation step to verify the certificate's SANs match the target infrastructure requirements.
4.  **Why did the outage last 2.5 hours?**
    *   The monitoring system was hosted on the same cluster as the failing application. When the cluster hit the pod limit, monitoring stopped, and no alerts were generated.
5.  **Why was the monitoring system on the same cluster?**
    *   To reduce infrastructure costs, observability tools were co-located with production workloads, creating a single point of failure for system visibility.

## 8. Conclusion
This incident highlighted the fragility of our infrastructure when a single component failure is allowed to propagate unchecked. While the initial certificate error was the spark, the cascading failures of the auto-scaler, cluster capacity, and monitoring system turned a 5-minute fix into a 2.5-hour outage. By decoupling our observability and adding validation gates, we can ensure that future incidents are detected immediately and contained effectively.
