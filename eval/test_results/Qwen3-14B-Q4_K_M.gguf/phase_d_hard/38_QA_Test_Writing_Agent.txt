Write tests for this pagination function. It has a SUBTLE BUG — your tests must catch it. 

```python
def paginate(items, page, per_page=10):
    """Return a page of items with metadata."""
    total = len(items)
    total_pages = total // per_page + (1 if total % per_page else 0)
    
    start = (page - 1) * per_page
    end = start + per_page
    
    return {
        "items": items[start:end],
        "page": page,
        "per_page": per_page,
        "total": total,
        "total_pages": total_pages,
        "has_next": page < total_pages,
        "has_prev": page > 1
    }
```

7. FIND THE BUG (hint: what happens with page=0? or negative page?)

One of your tests MUST fail, proving the bug exists. Explain what the bug is.

### 🔍 **Explanation of the Bug**

The bug lies in the **absence of validation for the `page` parameter**. The function allows `page=0` or even negative values, which are **invalid** in the context of pagination. For example:

- If `page=0`, the function calculates `start = (0 - 1) * per_page = -per_page`, which results in an **invalid slice** (`items[-per_page:per_page]`).
- The `has_next` flag is computed as `page < total_pages`, which for `page=0` and `total_pages=1` would return `True`, even though **page 0 is not a valid page number**.

This leads to **incorrect metadata** and **unexpected behavior** when invalid page numbers are passed to the function.

---

### ✅ **Conclusion**

The test for `page=0` is the only one that will fail, revealing the bug. The function should be updated to validate that `page` is a positive integer before proceeding with slicing or computing metadata.
